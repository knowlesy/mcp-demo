# MCP Demo: Claude Kubernetes Chaos Sandbox

A local Kubernetes sandbox purpose-built to test Claude's Model Context Protocol (MCP)
integrations. Deploy a full observability stack, trigger controlled chaos scenarios, and
watch Claude autonomously diagnose and resolve incidents using MCP tools.

---

## AI Context Snapshot (Repomix)

`repomix-output.xml` at the root of this repo is a **single-file snapshot of the entire
codebase** generated by [Repomix](https://github.com/yamadashy/repomix). Feed it directly
to any AI tool (Claude, GPT-4, Gemini) for instant full-codebase context without
requiring file-by-file access.

**Config:** [`repomix.config.json`](repomix.config.json) — controls which files are
included, output format (XML), and security checks.

**Keeping it current:**

- **Automatically** — a Claude Code `PostToolUse` hook in [`.claude/settings.json`](.claude/settings.json)
  detects every `git commit` Bash call and re-runs `repomix`, then stages
  `repomix-output.xml` as part of that commit.
- **Manually** — run the `/repomix` Claude Code skill (`.claude/commands/repomix.md`),
  or from the shell:
  ```bash
  repomix --config repomix.config.json
  ```

**Why it's committed:** Committing the snapshot means the AI context is always at the
same revision as the code. Any tool that clones this repo gets an immediately usable
context file without needing to run repomix locally.

## Quick Start

```bash
# 1. Start minikube with resource limits
make init

# 2. Deploy all infrastructure (Prometheus, Grafana, ArgoCD, Rundeck, Gitea)
make deploy-infra

# 3. Build and push dummy-webapp, configure ArgoCD to sync it
make deploy-apps

# 4. Add /etc/hosts entries (prints the command to run with sudo)
make hosts

# 5. Open all dashboards
make open
```

## Active MCPs and Their Roles

Claude must be configured with the following MCPs to run the agentic troubleshooting loop:

| MCP | Used For |
|-----|----------|
| **Kubernetes** | `kubectl get/describe/exec/scale/apply` — primary diagnosis tool |
| **Prometheus** | Query metrics (`rate`, `increase`, `histogram_quantile`) |
| **Grafana** | Read dashboard panels, annotate incidents |
| **ArgoCD** | Trigger sync, check app health, diff manifests |
| **Rundeck** | Trigger fix jobs, read job execution history |
| **Ansible** | Run playbooks directly for config fixes |
| **Filesystem** | Read SOPs from `docs/runbooks/` |
| **Git** | Update manifests, commit, push to Gitea |
| **Bash** | `curl`, `hey`, raw shell commands |
| **SQLite** | Inspect and repair the dummy-webapp database |
| **Slack** | Post incident summaries and resolution notices |
| **Memory** | Persist incident timeline across tool calls |

---

## Agentic Troubleshooting Loop

Claude is designed to follow this pattern when an incident fires:

```
1. DETECT    — Kubernetes MCP: get pod events, describe failing pod
2. READ SOP  — Filesystem MCP: read docs/runbooks/<scenario>.md
3. DIAGNOSE  — Prometheus/Grafana MCP: query relevant metrics
4. PLAN      — Memory MCP: store working hypothesis
5. FIX       — Git/ArgoCD/Ansible/Rundeck MCP: apply the fix
6. VERIFY    — Bash/Kubernetes MCP: confirm healthy state
7. REPORT    — Slack MCP: post incident summary
```

Each runbook in `docs/runbooks/` is written in machine-readable format with
explicit tool call suggestions so Claude can follow them step by step.
